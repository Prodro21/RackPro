# Phase 1.1: Device Outline MVP - Research

**Researched:** 2026-02-21
**Domain:** AI Vision outline extraction, SVG dimensional accuracy, on-demand asset loading
**Confidence:** MEDIUM

## Summary

This phase creates a pipeline to generate fabrication-accurate device silhouette outlines from product photos, store them as dimensioned SVGs, and load them on-demand in the app for visual rendering and snug-fit tray generation. The core challenge is NOT computer vision (device shapes are simple geometric forms) but dimensional accuracy: the outline must match the manufacturer bounding box within 0.5mm tolerance.

The recommended approach is a **hybrid pipeline**: use Claude Vision API to analyze a product photo and extract a simplified SVG path describing the device profile, then scale/validate that path against known bounding box dimensions from the catalog. Potrace (bitmap-to-vector tracing) was evaluated but rejected as the primary tool because product photos have complex backgrounds, reflections, and angles that make automated tracing unreliable without substantial pre-processing. Claude Vision can reason about the device shape and produce clean geometric outlines directly.

**Primary recommendation:** Build a Node.js CLI script (`scripts/generate-outline.ts`) that takes a product photo + device slug, calls Claude Vision API to extract simplified SVG path data, scales it to match catalog dimensions, validates accuracy, and writes `{slug}-{face}.svg` files. Load these SVGs in the app via `fetch()` with caching, parse path data, and render inline. Fall back to rectangular bounding box when no outline exists.

### Pattern 1: Outline SVG Format Convention
**What:** Each outline SVG uses a coordinate system where 1 unit = 1mm, matching the device's physical dimensions.
**When to use:** Always. This is the core dimensional accuracy contract.
**Example:**
```svg
<!-- usw-lite-16-poe-top.svg -->
<!-- viewBox matches device width x depth in mm -->
<svg xmlns="http://www.w3.org/2000/svg"
     viewBox="0 0 191.7 185"
     width="191.7mm" height="185mm">
  <metadata>
    <rackpro:outline
      slug="usw-lite-16-poe"
      face="top"
      bbox-w="191.7"
      bbox-d="185"
      tolerance="0.3"
      generated="2026-02-21"
      source="claude-vision" />
  </metadata>
  <path d="M 2 0 L 189.7 0 Q 191.7 0 191.7 2 L 191.7 183 Q 191.7 185 189.7 185 L 2 185 Q 0 185 0 183 L 0 2 Q 0 0 2 0 Z"
        fill="none" stroke="black" stroke-width="0.5" />
</svg>
```

**Key constraints:**
- `viewBox` origin at (0,0), width/height match device bounding box in mm
- Path coordinates in mm, fitting within the viewBox
- The path MUST fit within the bounding box and touch all four edges (within tolerance)
- Metadata block records provenance, tolerance, and generation date

### Pattern 3: On-Demand SVG Loading with Cache
**What:** Fetch outline SVGs lazily (only when a device is placed on the panel), cache in memory, fall back to rectangle.
**When to use:** In the app runtime (not the generation script).
**Example:**
```typescript
// src/catalog/outlines.ts
const outlineCache = new Map<string, string | null>();

export type DeviceFace = 'top' | 'front' | 'side';

export async function loadOutlinePath(
  slug: string,
  face: DeviceFace,
): Promise<string | null> {
  const key = `${slug}-${face}`;
  if (outlineCache.has(key)) return outlineCache.get(key)!;

  try {
    const res = await fetch(`/catalog/outlines/${key}.svg`);
    if (!res.ok) {
      outlineCache.set(key, null);
      return null;
    }
    const svgText = await res.text();
    const pathMatch = svgText.match(/d="([^"]+)"/);
    const pathData = pathMatch?.[1] ?? null;
    outlineCache.set(key, pathData);
    return pathData;
  } catch {
    outlineCache.set(key, null);
    return null;
  }
}
```

### Pattern 4: Rectangle Fallback in FrontView
**What:** When no outline SVG exists, render the existing rectangle representation.
**When to use:** Always -- most of the 50+ catalog devices will NOT have outlines initially.
**Example:**
```typescript
// In FrontView.tsx device rendering section
{outlinePath ? (
  <path
    d={outlinePath}
    transform={`translate(${ex}, ${ey}) scale(${SC})`}
    fill="#0d0d14"
    stroke={lib?.color ?? '#555'}
    strokeWidth={0.6}
  />
) : (
  <>
    <rect x={ex} y={ey} width={ew} height={eh}
          rx={1.5} fill="#0d0d14" stroke={lib?.color ?? '#555'}
          strokeWidth={0.6} />
    <line x1={ex} y1={ey} x2={ex+ew} y2={ey+eh}
          stroke="#222" strokeWidth={0.3} />
    <line x1={ex+ew} y1={ey} x2={ex} y2={ey+eh}
          stroke="#222" strokeWidth={0.3} />
  </>
)}
```

### Anti-Patterns to Avoid
- **Baking SVG data into the JS bundle:** Outline SVGs should be in `public/catalog/outlines/`, not imported into TypeScript. They are optional assets loaded on-demand.
- **Using pixel-based coordinate systems:** SVG viewBox MUST use mm units matching the device dimensions. Never use pixels -- there is no meaningful pixel-to-mm mapping for product photos.
- **Trusting Claude Vision output without validation:** The AI can produce paths that are dimensionally incorrect. Always validate that the generated path's bounding box matches the catalog dimensions within tolerance.
- **Blocking app render on outline fetch:** Outline loading must be async and non-blocking. Show rectangle immediately, swap to outline when loaded.
- **Storing outline data in the Zustand store:** Outlines are static assets, not user state. Use a module-level cache (like the catalog store pattern) to avoid unnecessary re-renders.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Image-to-vector tracing | Custom edge detection | `potrace` (as backup) or Claude Vision | Potrace implements the Bezier curve fitting algorithm from Peter Selinger's 2003 paper; reimplementing is error-prone |
| SVG path bounding box calculation | Manual path parsing | DOMParser + `getBBox()` in Node.js via `jsdom` or compute from path commands | SVG path commands (arcs, curves) have complex bounding box math |
| Image preprocessing | Custom pixel manipulation | `sharp` | ICC color profiles, EXIF rotation, alpha channel handling are subtle |
| SVG path scaling/translation | String regex replace on `d` attribute | Parse path commands, apply affine transform, serialize back | Regex on SVG path strings breaks on arcs, relative commands, shorthand curves |

**Key insight:** The outline generation script is a build-time tool, not a runtime dependency. It can afford heavier dependencies (sharp, AI SDK) because it runs on developer machines during asset creation, not in the browser.

## Common Pitfalls

### Pitfall 1: Product Photos Are Not Silhouettes
**What goes wrong:** Trying to auto-trace a product photo directly produces noisy, inaccurate outlines because of backgrounds, shadows, reflections, perspective distortion, and cables.
**Why it happens:** Product photos are designed for marketing, not dimensional measurement.
**How to avoid:** Use Claude Vision to reason about the shape and produce a simplified geometric outline, OR manually prepare clean silhouette images (white background, no shadows) before tracing. The AI approach is more scalable.
**Warning signs:** Generated outlines have hundreds of tiny path segments, jagged edges, or non-closed paths.

### Pitfall 2: Dimensional Drift After Scaling
**What goes wrong:** The outline looks right visually but is 1-3mm off from the catalog dimensions, causing trays to be too tight or too loose.
**Why it happens:** Claude Vision generates paths in an arbitrary coordinate space. If the scaling transformation has rounding errors or uses the wrong reference dimension, accuracy suffers.
**How to avoid:** After generating an outline path, compute its actual bounding box, then apply an affine scale to make the bbox match the catalog dimensions exactly. Validate with assertion: `Math.abs(pathBbox.width - catalogWidth) < 0.5`.
**Warning signs:** Tray generation reports interference or excessive gaps.

### Pitfall 3: SVG Path Coordinate System Confusion
**What goes wrong:** The outline renders at the wrong position, scale, or orientation in FrontView.
**Why it happens:** The FrontView uses panel-relative coordinates with a scale factor (`SC`). The outline SVG uses device-local coordinates (0,0 to W,H in mm). The transform chain must correctly map between them.
**How to avoid:** The outline path should be in device-local coordinates (0,0 origin, mm units). The FrontView applies `translate(elementX, elementY)` and `scale(SC)` when rendering. This matches how the existing rectangle rendering works -- the outline is just a replacement for the rect.

### Pitfall 4: Claude Vision Hallucinates Features
**What goes wrong:** Claude adds features to the outline that don't exist (phantom ports, wrong corner radii) or simplifies away important features (ventilation slots, cable recesses).
**Why it happens:** Claude Vision is an LLM, not a CAD measurement tool. It reasons about what it "sees" and may infer or miss details.
**How to avoid:** The generation script should include a human review step. Generate outline, display it overlaid on the original photo at correct dimensions, and ask the operator to approve or adjust. Also validate that the outline path area is between 70-100% of the bounding box area (devices are mostly rectangular).
**Warning signs:** Path area < 70% of bounding box (suggests excessive cutouts or errors), path self-intersects, or path has > 50 commands (over-detailed).

### Pitfall 5: CORS Issues Loading SVGs from public/
**What goes wrong:** `fetch('/catalog/outlines/...')` fails or returns HTML instead of SVG during development.
**Why it happens:** Vite dev server might not serve SVG files with the correct MIME type, or the path is wrong.
**How to avoid:** Vite serves files from `public/` at the root path by default. Verify with `curl http://localhost:5173/catalog/outlines/test.svg` during development. The fetch response should have `Content-Type: image/svg+xml`.

## Code Examples

### Claude Vision API Call for Outline Extraction
```typescript
// scripts/generate-outline.ts
import Anthropic from '@anthropic-ai/sdk';
import * as fs from 'node:fs';

const client = new Anthropic();

async function extractOutline(
  imagePath: string,
  face: 'top' | 'front' | 'side',
  bboxW: number,
  bboxH: number,
): Promise<string> {
  const imageData = fs.readFileSync(imagePath);
  const base64 = imageData.toString('base64');
  const mediaType = imagePath.endsWith('.png') ? 'image/png' : 'image/jpeg';

  const response = await client.messages.create({
    model: 'claude-sonnet-4-5-20250514',
    max_tokens: 4096,
    messages: [{
      role: 'user',
      content: [
        {
          type: 'image',
          source: { type: 'base64', media_type: mediaType, data: base64 },
        },
        {
          type: 'text',
          text: `You are analyzing a ${face} view photo of a network device.
The device bounding box is ${bboxW}mm wide x ${bboxH}mm tall.

Generate an SVG path that traces the OUTER SILHOUETTE of the device body.
Use a coordinate system where (0,0) is top-left and the viewBox is "0 0 ${bboxW} ${bboxH}".

Rules:
- The path MUST be a single closed path (ending with Z)
- Use absolute coordinates only (M, L, Q, C, A commands — no lowercase)
- Include rounded corners where visible (use Q or A commands)
- Simplify: ignore ports, LEDs, labels, ventilation holes — only the outer body shape
- The path bounding box must be exactly ${bboxW} x ${bboxH}
- Use at most 30 path commands — this is a simplified outline, not a detail trace

Return ONLY the SVG path d="" attribute value, nothing else.`,
        },
      ],
    }],
  });

  const text = response.content[0];
  if (text.type !== 'text') throw new Error('Unexpected response type');
  return text.text.trim();
}
```

### Outline Validation Function
```typescript
// scripts/generate-outline.ts (continued)
function validateOutlinePath(
  pathD: string,
  expectedW: number,
  expectedH: number,
  tolerance: number = 0.5,
): { valid: boolean; actualW: number; actualH: number; error?: string } {
  // Parse path commands to extract all x,y coordinates
  const coords: { x: number; y: number }[] = [];
  const cmdRegex = /([MLHVCSQTAZ])\s*([\d.\s,-]*)/gi;
  let match;
  let cx = 0, cy = 0;

  while ((match = cmdRegex.exec(pathD)) !== null) {
    const cmd = match[1].toUpperCase();
    const nums = match[2].trim().split(/[\s,]+/).map(Number).filter(n => !isNaN(n));

    switch (cmd) {
      case 'M': case 'L':
        for (let i = 0; i < nums.length - 1; i += 2) {
          cx = nums[i]; cy = nums[i + 1];
          coords.push({ x: cx, y: cy });
        }
        break;
      case 'H': cx = nums[0]; coords.push({ x: cx, y: cy }); break;
      case 'V': cy = nums[0]; coords.push({ x: cx, y: cy }); break;
      case 'Q':
        for (let i = 0; i < nums.length - 3; i += 4) {
          coords.push({ x: nums[i], y: nums[i + 1] });
          cx = nums[i + 2]; cy = nums[i + 3];
          coords.push({ x: cx, y: cy });
        }
        break;
      case 'C':
        for (let i = 0; i < nums.length - 5; i += 6) {
          coords.push({ x: nums[i], y: nums[i + 1] });
          coords.push({ x: nums[i + 2], y: nums[i + 3] });
          cx = nums[i + 4]; cy = nums[i + 5];
          coords.push({ x: cx, y: cy });
        }
        break;
    }
  }

  if (coords.length === 0) return { valid: false, actualW: 0, actualH: 0, error: 'No coordinates found' };

  const xs = coords.map(c => c.x);
  const ys = coords.map(c => c.y);
  const minX = Math.min(...xs), maxX = Math.max(...xs);
  const minY = Math.min(...ys), maxY = Math.max(...ys);
  const actualW = maxX - minX;
  const actualH = maxY - minY;

  const wErr = Math.abs(actualW - expectedW);
  const hErr = Math.abs(actualH - expectedH);

  if (wErr > tolerance || hErr > tolerance) {
    return {
      valid: false, actualW, actualH,
      error: `Dimension mismatch: got ${actualW.toFixed(1)}x${actualH.toFixed(1)}, expected ${expectedW}x${expectedH} (delta W=${wErr.toFixed(2)}, H=${hErr.toFixed(2)}, tolerance=${tolerance})`,
    };
  }

  return { valid: true, actualW, actualH };
}
```

### SVG File Writer
```typescript
function writeOutlineSvg(
  slug: string,
  face: 'top' | 'front' | 'side',
  pathD: string,
  bboxW: number,
  bboxH: number,
  outputDir: string,
): void {
  const svg = `<?xml version="1.0" encoding="UTF-8"?>
<svg xmlns="http://www.w3.org/2000/svg"
     viewBox="0 0 ${bboxW} ${bboxH}"
     width="${bboxW}mm" height="${bboxH}mm">
  <!-- Generated by RackPro outline generator -->
  <!-- Device: ${slug}, Face: ${face} -->
  <!-- BBox: ${bboxW} x ${bboxH} mm -->
  <path d="${pathD}" fill="none" stroke="black" stroke-width="0.5" />
</svg>`;

  const filename = `${slug}-${face}.svg`;
  fs.writeFileSync(`${outputDir}/${filename}`, svg);
}
```

## Open Questions

1. **Which faces are most valuable?**
   - What we know: The phase specifies top, front, and side faces. Front face is used in FrontView rendering. Top face is used for tray generation (the shape that sits in the tray). Side face is used in SideView.
   - What's unclear: Whether the user needs all three faces for the initial 5-10 devices, or if starting with just the `front` face (for visual accuracy in FrontView) is sufficient for MVP.
   - Recommendation: Prioritize `top` face (drives tray geometry) and `front` face (drives visual rendering). `side` face can be deferred to batch 2.

2. **Image sourcing for 5-10 priority devices**
   - What we know: Ubiquiti publishes product photos on techspecs.ui.com and the Ubiquiti Store. These are high-quality PNG/JPEG images with white backgrounds.
   - What's unclear: Which 5-10 devices the user considers "priority." Best candidates for demonstration are compact non-rack-width devices where outline differs most from a rectangle.
   - Recommendation: Start with devices that are NOT rectangular boxes: UX7 (rounded cube), USW-Flex (compact with rounded corners), USW-Lite-16-PoE (slightly rounded), Raspberry Pi 5 (board with irregular edges), ATEM Mini Pro (control surface). These show the most visual value from outlines.

3. **How outlines flow to export generators**
   - What we know: OpenSCAD and Fusion 360 generators currently use `el.w` and `el.h` for device cutout dimensions. For snug-fit trays, they need the actual device profile (top-view outline) rather than a rectangle.
   - What's unclear: Whether the outline path should be included in the ExportConfig JSON or loaded separately by each generator.
   - Recommendation: For the MVP, include an optional `outlinePath` field in ExportElement. Generators that support outlines (Fusion 360) can use `linear_extrude(outline_path)` for tray floor cutouts. Generators that don't (OpenSCAD via BOSL2) continue using rectangular dimensions. This avoids blocking the MVP on export pipeline changes.

4. **API cost for Claude Vision calls**
   - What we know: Claude Sonnet 4.5 pricing applies to vision API calls. Each image analysis uses input tokens for the image (varies by resolution) plus output tokens for the path.
   - What's unclear: Exact cost per outline generation call.
   - Recommendation: Resize images to 1024px longest edge before sending (reduces token cost without meaningful quality loss for silhouette extraction). Estimate ~$0.01-0.05 per outline, or $0.30-1.50 for all 30 outlines (10 devices x 3 faces). Acceptable for a one-time generation tool.
