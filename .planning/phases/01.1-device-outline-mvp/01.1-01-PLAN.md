  <action>
Create `scripts/generate-outline.ts` -- a CLI tool that generates a dimensioned SVG outline from a product photo using Claude Vision API.

**CLI interface (using commander):**
```
npx tsx scripts/generate-outline.ts \
  --slug usw-lite-16-poe \
  --face top \
  --image path/to/photo.jpg \
  [--output public/catalog/outlines] \
  [--tolerance 0.5] \
  [--model claude-sonnet-4-5-20250514] \
  [--dry-run]
```

Options:
- `--slug` (required): Device slug from catalog. Script reads `public/catalog/devices.json` to get bounding box dimensions.
- `--face` (required): One of `top`, `front`, `side`.
- `--image` (required): Path to product photo (JPEG or PNG).
- `--output` (default: `public/catalog/outlines`): Output directory for SVGs.
- `--tolerance` (default: `0.5`): Max allowed dimensional deviation in mm.
- `--model` (default: `claude-sonnet-4-5-20250514`): Anthropic model to use.
- `--dry-run`: Print the prompt and exit without calling the API.

**Implementation steps:**

1. **Parse CLI args** with commander. Validate slug exists in devices.json by reading and JSON-parsing `public/catalog/devices.json` (do NOT import from src/ -- scripts run standalone).

2. **Determine bounding box** based on face:
   - `top`: width x depth
   - `front`: width x height
   - `side`: depth x height

3. **Preprocess image** with sharp: resize to max 1536px on longest edge (reduces API token cost), convert to PNG, get base64.

4. **Call Claude Vision API** using `@anthropic-ai/sdk`. The prompt must:
   - Identify the face being analyzed (top/front/side)
   - Specify the bounding box in mm
   - Request a SINGLE CLOSED SVG path (ending with Z)
   - Request ABSOLUTE coordinates only (M, L, Q, C, A -- no lowercase)
   - Request simplified outline: outer body silhouette only, ignore ports/LEDs/labels/vents
   - Request max 30 path commands
   - Request path bounding box to be exactly bboxW x bboxH
   - Use the prompt structure from RESEARCH.md code example

5. **Parse and validate the response:**
   - Extract the path `d` attribute from the response text (strip any markdown code fences if present)
   - Compute the actual bounding box of the path by parsing M/L/H/V/Q/C commands and extracting all x,y coordinates
   - Compare actual bbox dimensions to expected dimensions
   - If outside tolerance, attempt ONE correction: compute scale factors `(expectedW / actualW)` and `(expectedH / actualH)`, apply affine scale to all coordinates in the path, re-validate
   - If still outside tolerance after correction, print error and exit(1)

6. **Write SVG file:**
   - Format: `{slug}-{face}.svg`
   - viewBox: `0 0 {bboxW} {bboxH}`
   - width/height attributes in mm units
   - XML comment with device slug, face, generation date
   - Single `<path>` element with the validated/scaled path data
   - `fill="none" stroke="black" stroke-width="0.5"`

7. **Print summary:** slug, face, bbox, actual path bbox, tolerance delta, output file path.

**Error handling:**
- Missing ANTHROPIC_API_KEY env var: print clear error message with instructions
- Image file not found: print error and exit(1)
- Slug not found in catalog: print available slugs and exit(1)
- API error: print error details and exit(1)
- Validation failure after correction attempt: print dimensional mismatch details and exit(1)

**Path parsing function** (reusable, export it):
Create and export `parseSvgPathBounds(d: string): { minX: number, minY: number, maxX: number, maxY: number, width: number, height: number }` that handles M, L, H, V, Q, C, A (absolute) commands. For Q (quadratic Bezier), include both control point and endpoint. For C (cubic Bezier), include both control points and endpoint. Skip Z commands. Return min/max x/y and derived width/height.

**Path scaling function** (reusable, export it):
Create and export `scaleSvgPath(d: string, scaleX: number, scaleY: number): string` that parses path commands, multiplies all x-coordinates by scaleX and y-coordinates by scaleY, and serializes back. Handle M, L, H, V, Q, C, A commands correctly (H only has x, V only has y, A has rx/ry/rotation/flags/x/y).

Do NOT use `svg-path-parse` library -- the path format from Claude Vision is simple enough that a regex-based parser handles it correctly (per RESEARCH.md recommendation against hand-rolling only applies to complex path manipulation, not this simple coordinate extraction).
  </action>

  <action>
Create `scripts/validate-outlines.ts` -- a CLI tool that validates all existing outline SVGs against catalog dimensions.

**CLI interface:**
```
npx tsx scripts/validate-outlines.ts \
  [--dir public/catalog/outlines] \
  [--tolerance 0.5]
```

**Implementation:**

1. **Scan directory** for `*.svg` files matching the `{slug}-{face}.svg` naming convention.

2. **For each SVG file:**
   - Parse filename to extract slug and face
   - Look up device in `public/catalog/devices.json` -- skip if slug not found (warn)
   - Determine expected bbox dimensions based on face (same logic as generate-outline)
   - Read SVG file, extract the `d` attribute from the `<path>` element using regex
   - Compute path bounding box using the `parseSvgPathBounds` function (import from generate-outline.ts)
   - Also validate viewBox matches expected dimensions
   - Compare actual vs expected dimensions

3. **Report results** in a table format:
   ```
   File                          Expected     Actual       Delta    Status
   usw-lite-16-poe-top.svg       191.7x185    191.7x185    0x0      PASS
   ux7-front.svg                 117x42.5     117.2x42.5   0.2x0    PASS
   bad-device-side.svg           100x50       97x48        3x2      FAIL
   ```

4. **Exit code:** 0 if all pass, 1 if any fail.

Import `parseSvgPathBounds` from `./generate-outline.ts` (both scripts share this utility). If this creates a circular dependency concern, extract the shared function into `scripts/lib/svg-path-utils.ts` and import from there in both scripts.

**Additional checks:**
- Path must be closed (contains `Z` or `z`)
- Path must have at least 4 commands (minimum for a closed shape)
- Path area should be between 50-100% of bounding box area (heuristic check for obviously wrong outlines). Approximate area using the shoelace formula on the extracted coordinates.
  </action>

  </verify>
